AutoreleasePool源码解析

AutoreleasePool结构

大致看下AutoreleasePoolPage的结构可以知道：这个AutoreleasePool是一个以AutoreleasePoolPage为节点的双向链表。每一个AutoreleasePoolPage节点大小是4096字节，并对应一个线程。

AutoreleasePool构造函数和方法

说明：判断节点是否为空，其实就是判断当前next是否指向的起始位置（begin())。判断节点是否满了，那么next是否指向page最大位置（end())。

hotPage

hotPage表示当前的autoreleasePoolPage,是存在TLS里面的。这里不得不提到一个概念TLS（线程局部存储, Thread Local Storage）。

线程局部存储为了避免多个线程同时访存同一全局变量或者静态变量时所导致的冲突，尤其是多个线程同时需要修改这一变量时。为了解决这个问题，我们可以通过TLS机制，为每一个使用该全局变量的线程都提供一个变量值的副本，每一个线程均可以独立地改变自己的副本，而不会和其它线程的副本冲突。从线程的角度看，就好像每一个线程都完全拥有该变量。而从全局变量的角度上来看，就好像一个全局变量被克隆成了多份副本，而每一份副本都可以被一个线程独立地改变。

AutoreleasePoolPage::push()

push执行的时候先判断是否需要每个pool都生成一个新的page（DebugPoolAllocation）,
是则执行autoreleaseNewPage方法，否则执行autoreleaseFast方法。传入哨兵标记POOL_BOUNDARY，这是一个边界标记。

iOS编译原理

Clang编译到LLVM编译的过程

1.预处理

头文件引入，宏替换，注释处理，条件编译（#ifdef)等操作

2.词法分析

词法分析器读入源文件的字符流，产生词法单元（Token）作为输出

3.语法分析

Token流会被解析成一颗抽象语法树（AST）

4.CodeGen

遍历语法树，生成LLVM IR代码

5.生成汇编代码

LLVM对IR进行优化，会针对不同的架构生成不同的目标代码，以汇编代码输出

6.汇编器

将汇编代码转为机器语言，最后输出目标文件

7.链接

将编译产生的.o文件和dylib、a文件，生成一个mach-o文件

编译和启动加载的一些概念汇总

App启动的过程

  1. 解析info.plist，加载相关信息，建立沙盒和权限检查
  2. Mach-O加载
  3. 调用main(), UIApplicationMain(), applicationWillFinishLaunching

Mach-O流程（pre-main）

  • 加载动态库dyld。dyld先读取mach-o的header和load commands，检查其依赖的动态库，递归加载，大多都是系统的动态库，会被缓存到dyld shared cache，提高读取效率。
  • Rebase && Bind。处于安全的考虑，ASLR使得地址随机化，导致起始地址不固定，Rebase修正内部（指向当前mach-o文件）的指针指向， Bind修正外部指针指向
    ，Rebase&&Bind用来指向正确的地址，Objective-C 中有很多数据结构都是靠 Rebasing 和 Binding 来修正（fix-up）的，比如 Class 中指向父类的指针和指向方法的指针。
  • ObjC SetUp。读取二进制文件中Data段内容，找到objc相关信息；注册Objc类，ObjC Runtime 需要维护一张映射类名与类的全局表。当加载一个 dylib 时，其定义的所有的类都需要被注册到这个全局表中；读取 protocol 以及 category 的信息，把category的定义插入方法列表 (category registration)；确保 selector 的唯一性
  • Initializers。动态调整，在堆和栈中写入内容。
    • Objc的+load()函数
    • C++的构造函数属性函数 形如attribute
    • dyld开始将程序二进制文件初始化
    • 交由ImageLoader读取image，其中包含了我们的类、方法等各种符号
    • runtime接手后调用mapimages做解析和处理，接下来loadimages中调用 callloadmethods方法，遍历所有加载进来的Class，按继承层级依次调用Class的+load方法和其 Category的+load方法

ASLR地址空间布局随机化

如果采用ASLR，进程每次启动，地址空间都会被简单地随机化，防止黑客攻击

Code Sign

代码签名会直接写入到mach-o的可执行文件里，加密哈希不是针对于整个文件，而是针对于每一个Page的。这就保证了在dyld进行加载的时候，可以对每一个page进行独立的验证。

mach-o文件的组成

  • Header头部：包含可以执行的CPU架构，比如x86、arm64
  • Load commands加载命令：包含文件的组织架构和在虚拟内存中的布局方式
  • Data数据，包含load commands中需要的各个段(segement）数据，每个Segement都是Page的整数倍。每个 segement 又被划分成一些 section。segment 的名字都是大写的，且空间大小为页的整数。页的大小跟硬件有关，在 arm64 架构一页是 16KB，其余为 4KB。

Data部分包含三个段

  • _TEXT代码段，readonly,包含函数和只读字符串
  • _DATA数据段，readwrite,包含可读写全局变量
  • _LINKEDIT 包含方法和变量的位置、偏移量

回溯法的定义

一个问题解决的可能性都可以抽象为一个树状结构，在某一步含有n个可能的选项，那么这一步即可形象的表示为树中的一个节点，这个节点有n个子节点。遍历这个树，叶节点包含答案，则顺着叶节点的子节点寻找下去，否则，回溯到该叶节点的上一个节点去尝试其他节点，直至寻找到答案为止。

如何使用回溯法

eg:给定一个数组[1,2,3,4],求可能的排列顺序的可能性。

iOS探究Block的实现

Block的类型

  • NSGlobalBlock 位于text段
  • NSStackBlock 位于栈内存
  • NSMallocBlock 位于堆内存
  1. NSGlobalBlock是因为没有引用任何外部变量，block不需要建立局部变量值的快照。
  2. NSStackBlock使用了局部变量，局部变量的值被copy到栈上，以供block使用。但实际上很多场景都不是NSStackBlock，而是NSMallocBlock。这是因为ARC下编译器在NSStackBlock类型的block传递过程中进行了自动优化，将传递过程将 block拷贝到堆。分析见这篇文章ARC下NSStackBlock去哪了。
  3. NSMallocBlock 拷贝block到堆上。

__block的使用

Block内引用外部变量a是只读的，但是如果想在block内部改变a时，需要在定义a的时候使用__block修饰。

iOS图片加载过程以及优化

UIImageView加载图片的过程

  1. 从磁盘拷贝数据到内核缓冲区域；

  2. 从内核缓存区域复制数据到用户内存；

  3. UIImage赋值给UIImageView的image时，图像数据会被解码，变成位图数，内存大小为 width height 4bytes （4bytes:每个像素点的大小) 。

  4. CTTransaction捕捉到UIImageView layer树的变化；

  5. 主线程 runloop提交CTTransaction，开始图像渲染。（如果数据没有字节对齐，Core Animation会再拷贝一份数据，进行字节对齐，GPU处理位图数据，进行渲染）。

UIImage几种加载方式的区别

  • imageNamed：在图片第一次渲染到屏幕的时候触发解码，缓存解码之后的数据，缓存在全局内存中，不会随着UIImage的释放而释放。

  • imageWithContentsOfFile: 或 imageWithData: 与imageNamed不同的是会随着UIImage的释放而释放。

几种优化思路

降低采样率

即减少加载UIImage内存的大小,根据imageview实际size来加载，可以减少内存占用。解码后生成CGImage缩略图，再转化为UIImage,然后传给UIImgaeView渲染展示。

异步解码

解码的操作在主线程，比较耗费CPU的资源。可以把耗时的解码操作放入子线程，解码完成之后再回调到主线程刷新，例如SDWebImage。还有更加极限的优化是在子线程解码之后，将解码之后的图片存在磁盘之中，例如FastImageCache。

mmap内存映射

上面提到过从磁盘读取数据时，需要把数据读入内核缓冲区，然后复制到用户内存空间，是比较耗时的。mmap是把文件映射到用户空间的虚拟内存，等同于把文件放入内存，操作中并不消耗物理内存，效率比较高。

字节对齐

UIImageView在渲染之前对于未字节对齐的图像数据拷贝一份，然后进行字节对齐的操作，这一过程消耗也比较多。可以统一管理和优化字节对齐的时机。FastImageCache也针对字节对齐做了优化管理。具体将在下文重点分析FastImageCache。

swift用Stack实现一个队列

基本思路

栈是先进后出，而队列是先进先出，完全不同的顺序，如何做到用栈来实现队列呢？初步的思路是，将栈内容完全反转，这里我们需要两个栈，栈A为原始数据，然后反序压入栈B，再从栈B压出去，那么对于栈A来说数据就是先进先出了

代码实现

首先实现一个stack，定义压栈push、出栈pop、获取内容个数的size等接口。

入队操作，在A栈中压进item。出队操作，在B栈为空的情况下，把A栈压出的item压入B栈，那么位于A栈顶的item将位于B栈底，从而实现AB的顺序反转，如果B栈不为空，则按顺序压出item。从而实现A栈中原始数据的先进先出。
