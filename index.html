1.1 OS X和Darwin的区别

大家可能会混淆OS X和Darwin的定义和关系。OS X是整个操作系统的一个集体名称，Darwin是其中一个组件。Darwin是操作系统的类UNIX核心，本身由内核（kernel）、XNU和运行时组成。

一、OS X和iOS的架构

1.1 UNIX的系统目录

OS X是一个符合UNIX标准的系统。

  • /bin:UNIX二进制程序，常用UNIX命令（ls、mv）所在的目录
  • /sbin: 系统程序，用于系统管理，例如文件系统管理和网络配置
  • /usr: 第三方软件可以安装在这里
  • /etc: 其他文件。包含大部分系统的配置文件，例如密码文件（/etc/password)
  • /dev: BSD设备文件，这些特殊文件表示了系统中存在的设备（字符设备和块设备）
  • /tmp: 临时目录。唯一一个所有人都能读写的目录
  • /var: 各种杂项文件。日志文件、邮件存储等

1.2 XNU的概述

内核XNU是Darwin的核心，也是整个OSX的核心，由如下几个组件组成：

  • 1.Mach微内核
  • 2.BSD层
  • 3.libKern
  • 4.I/O Kit

mach 是XNU的核心和基础，提供进程和线程的抽象、虚拟内存管理、任务调度、进程间通信和消息传递。

BSD建立在Mach之上，也是XNU中不可分割的部分。这是一个可靠更加现代的API，提供更高层次的抽象，包括：

  • UNIX进程模型
  • POSIX线程模型和同步原语
  • UNIX用户和组
  • 网络协议栈（BSD Socket API)
  • 文件系统访问
  • 设备访问

libKern 内建的、自包含的C++库。

I/O Kit 内核中完整的、自包含执行环境，让开发者可以快速创建稳定优雅的设备驱动程序。

二、OS X和iOS使用的技术

2.1 BSD相关的特性

XNU向用户态展现出来的主要接口是BSD接口

2.1.1 sysctl

访问内核内部状态的标准方法，通过使用这条命令，系统管理员可以直接查询内核变量的值，获得重要的运行时诊断信息。

2.1.2 kqueue

BSD中使用的内核事件通知机制。一个kqueue指的是一个描述符，这个描述符会一直阻塞等待直到一个特定类型的事件发生。kqueue和kevet构成了内核异步I/O的基础.

2.1.3 强制访问控制 MAC（Mandatory Access Control)

允许更为精细的访问模型，添加了对象级别的安全性。比如通过这种方式，可以控制一个给定的应用程序不允许访问用户的私有数据和网站。
MAC是一个框架，提供一个稳固的基础，可以添加组件，并不要求是内核的一部分。可以通过插件的方式对系统的安全性进行控制，可以在MAC中注册内核的扩展，让这总内核扩展负责实施某一个安全策略。从内核的角度来看，在各种系统调用实现中插入对MAC的调用，每一个系统调用必须通过MAC的验证，然后才能真正处理来自用户态请求。
MAC是OS X的沙盒机制和IOS的entitlement机制的基础

2.2 OSX和IOS的安全机制

2.2.1 代码签名

在确认软件的安全性之前，必须对软件的来源进行验证。使用SSL验证网站的身份，鼓励开发者对自己开发的应用程序进行签名并通过这种方式验证身份。

2.2.2 沙盒机制

  • 无法突破应用程序目录之外的位置
  • 无法访问其他进程
  • 无法直接使用任何硬件设备，只能通过苹果提供的框架
  • 无法动态生成代码

2.2.3 iOS的entitlement

entitlement plist文件嵌入应用程序的二进制文件中，并由苹果进行数字签名。任务的相关条目允许在底层访问mach任务，mach任务支持BSD进程的底层内核原语。获得任务的端口就相当于拥有了任务，可以访问这个任务的虚拟内存和各种描述符。另一个重要的entitlement就是dynamic-codesigning,允许动态生成代码，据目前了解，只有mobilesafari才有这种entitlement。

二、Mach-O、进程和线程

进程的生命周期

进程的生命周期开始于SIDL状态，表示一个暂时的空闲状态，进程刚被父进程通过fork创建出来就处于这个状态。正在执行时，处于SRUN状态。正在运行的进程的时间片被用完了，或者被一个更高优先级的进程抢占了，那么这个处于正在运行状态的进程就会被踢出CPU并放回运行队列。

可执行文件

魔数(magic):操作系统内核讲文件读入内存，寻找一个头签名，据此可以确定精确的可执行格式。
通用二进制：基本思想是提供一个能够在任意架构上执行的完全可移植的二进制格式。调用一个二进制文件时，Mach加载器首先解析胖二进制文件头，确定其中可用的架构，只加载最合适的架构的代码。因此不相关的架构的代码不会占用任何内存。

Mach-O的二进制文件：包含mach-header、加载命令、Data数据段

mach-header结构如下

加载命令 (Load Commands)

这些指令详细的指导如何设置和加载二进制数据。部分是内核加载器直接使用，部分是动态链接器使用。加载过程在内核的部分负责新进程的基本设置——分配虚拟地址、创建主线程，代码签名/加密工作。对于动态链接的可执行过文件，真正的库加载和符号解析是通过LC_LOAD_DYLINKER命令指定的动态链接器在用户态完成。

  • LC_SEGMENT：最主要的加载命令。指导内核如何设置新运行的进程的内存空间。对于每一个段，将文件中相应的内容加载到内存中：从偏移量为fileoff处加载filesize字节到虚拟内存地址vmaddr处的vmsize字节。每一个段的页面都根据initprot进行初始化，initprot指定如何通过读/写/执行位初始化页面的保护级别。段的保护设置可以动态改变。
  • LC_UNIXTHREAD：当所有库都完成加载之后，LC_UNIXTHREAD负责启动二进制程序的主线程。罗列所有初始化寄存器的状态，大部分寄存器会初始化为0，其中指令或者程序计数器是例外，这些寄存器保存了程序入口的地址。
  • LC_MAIN 代替了LC_UNIXTHREAD，设置程序主线程的入口地址和栈大小。
  • LC_CODE_SIGNATURE 包含了Mach-O二进制文件的代码签名，如果这个签名和代码本身不匹配，那么内核会给进程发送一个SIGKILL信号杀死进程。

动态库

符号绑定（binding），mach-O镜像中有很多空间预留给了对外部的库和符号的引用，需要在程序启动时填充，这个过程由动态链接器来完成。

AutoreleasePool结构

大致看下AutoreleasePoolPage的结构可以知道：这个AutoreleasePool是一个以AutoreleasePoolPage为节点的双向链表。每一个AutoreleasePoolPage节点大小是4096字节，并对应一个线程。

AutoreleasePool构造函数和方法

说明：判断节点是否为空，其实就是判断当前next是否指向的起始位置（begin())。判断节点是否满了，那么next是否指向page最大位置（end())。

hotPage

hotPage表示当前的autoreleasePoolPage,是存在TLS里面的。这里不得不提到一个概念TLS（线程局部存储, Thread Local Storage）。

线程局部存储为了避免多个线程同时访存同一全局变量或者静态变量时所导致的冲突，尤其是多个线程同时需要修改这一变量时。为了解决这个问题，我们可以通过TLS机制，为每一个使用该全局变量的线程都提供一个变量值的副本，每一个线程均可以独立地改变自己的副本，而不会和其它线程的副本冲突。从线程的角度看，就好像每一个线程都完全拥有该变量。而从全局变量的角度上来看，就好像一个全局变量被克隆成了多份副本，而每一份副本都可以被一个线程独立地改变。

iOS编译原理

Clang编译到LLVM编译的过程

1.预处理

头文件引入，宏替换，注释处理，条件编译（#ifdef)等操作

2.词法分析

词法分析器读入源文件的字符流，产生词法单元（Token）作为输出

3.语法分析

Token流会被解析成一颗抽象语法树（AST）

4.CodeGen

遍历语法树，生成LLVM IR代码

5.生成汇编代码

LLVM对IR进行优化，会针对不同的架构生成不同的目标代码，以汇编代码输出

6.汇编器

将汇编代码转为机器语言，最后输出目标文件

7.链接

将编译产生的.o文件和dylib、a文件，生成一个mach-o文件

编译和启动加载的一些概念汇总

App启动的过程

  1. 解析info.plist，加载相关信息，建立沙盒和权限检查
  2. Mach-O加载
  3. 调用main(), UIApplicationMain(), applicationWillFinishLaunching

Mach-O流程（pre-main）

  • 加载动态库dyld。dyld先读取mach-o的header和load commands，检查其依赖的动态库，递归加载，大多都是系统的动态库，会被缓存到dyld shared cache，提高读取效率。
  • Rebase && Bind。处于安全的考虑，ASLR使得地址随机化，导致起始地址不固定，Rebase修正内部（指向当前mach-o文件）的指针指向， Bind修正外部指针指向
    ，Rebase&&Bind用来指向正确的地址，Objective-C 中有很多数据结构都是靠 Rebasing 和 Binding 来修正（fix-up）的，比如 Class 中指向父类的指针和指向方法的指针。
  • ObjC SetUp。读取二进制文件中Data段内容，找到objc相关信息；注册Objc类，ObjC Runtime 需要维护一张映射类名与类的全局表。当加载一个 dylib 时，其定义的所有的类都需要被注册到这个全局表中；读取 protocol 以及 category 的信息，把category的定义插入方法列表 (category registration)；确保 selector 的唯一性
  • Initializers。动态调整，在堆和栈中写入内容。
    • Objc的+load()函数
    • C++的构造函数属性函数 形如attribute
    • dyld开始将程序二进制文件初始化
    • 交由ImageLoader读取image，其中包含了我们的类、方法等各种符号
    • runtime接手后调用mapimages做解析和处理，接下来loadimages中调用 callloadmethods方法，遍历所有加载进来的Class，按继承层级依次调用Class的+load方法和其 Category的+load方法

ASLR地址空间布局随机化

如果采用ASLR，进程每次启动，地址空间都会被简单地随机化，防止黑客攻击

Code Sign

代码签名会直接写入到mach-o的可执行文件里，加密哈希不是针对于整个文件，而是针对于每一个Page的。这就保证了在dyld进行加载的时候，可以对每一个page进行独立的验证。

mach-o文件的组成

  • Header头部：包含可以执行的CPU架构，比如x86、arm64
  • Load commands加载命令：包含文件的组织架构和在虚拟内存中的布局方式
  • Data数据，包含load commands中需要的各个段(segement）数据，每个Segement都是Page的整数倍。每个 segement 又被划分成一些 section。segment 的名字都是大写的，且空间大小为页的整数。页的大小跟硬件有关，在 arm64 架构一页是 16KB，其余为 4KB。

Data部分包含三个段

  • _TEXT代码段，readonly,包含函数和只读字符串
  • _DATA数据段，readwrite,包含可读写全局变量
  • _LINKEDIT 包含方法和变量的位置、偏移量

数据结构：回溯法

回溯法的定义

一个问题解决的可能性都可以抽象为一个树状结构，在某一步含有n个可能的选项，那么这一步即可形象的表示为树中的一个节点，这个节点有n个子节点。遍历这个树，叶节点包含答案，则顺着叶节点的子节点寻找下去，否则，回溯到该叶节点的上一个节点去尝试其他节点，直至寻找到答案为止。

如何使用回溯法

eg:给定一个数组[1,2,3,4],求可能的排列顺序的可能性。

iOS探究Block的实现

Block的类型

  • NSGlobalBlock 位于text段
  • NSStackBlock 位于栈内存
  • NSMallocBlock 位于堆内存
  1. NSGlobalBlock是因为没有引用任何外部变量，block不需要建立局部变量值的快照。
  2. NSStackBlock使用了局部变量，局部变量的值被copy到栈上，以供block使用。但实际上很多场景都不是NSStackBlock，而是NSMallocBlock。这是因为ARC下编译器在NSStackBlock类型的block传递过程中进行了自动优化，将传递过程将 block拷贝到堆。分析见这篇文章ARC下NSStackBlock去哪了。
  3. NSMallocBlock 拷贝block到堆上。

__block的使用

Block内引用外部变量a是只读的，但是如果想在block内部改变a时，需要在定义a的时候使用__block修饰。
